# 緊急時の対応マニュアル

## 1. 概要

セキュリティインシデント発生時の緊急対応手順を説明します。不正アクセス、データ漏洩、アカウント乗っ取りなど、重大なセキュリティ問題への対処法を網羅しています。

**このマニュアルで解決できること**:
- 不正アクセス検知時の初動対応
- アカウント乗っ取りへの対処
- データ漏洩時の対応フロー
- パスワード漏洩時の処置
- 監査ログによる原因調査

**対象者**: Admin（管理者）、すべてのユーザー

**重要**: このマニュアルは緊急時用です。平時に一読しておくことを強く推奨します。

---

## 2. セキュリティインシデントの種類

### 2-1. インシデントの重要度分類

| レベル | 種類 | 例 | 初動対応時間 |
|--------|------|-----|------------|
| **🔴 緊急** | システム全体に影響 | 大規模なデータ漏洩、ランサムウェア | 即座（5分以内） |
| **🟠 重大** | 組織に重大な影響 | 管理者アカウント乗っ取り、不正ログイン | 即座（15分以内） |
| **🟡 警戒** | 一部に影響 | 一般アカウント乗っ取り、パスワード漏洩 | 30分以内 |
| **🟢 軽微** | 限定的な影響 | ログイン失敗の増加、疑わしい操作 | 1時間以内 |

---

## 3. 不正アクセス検知時の対応

### 3-1. 不正アクセスの兆候

**ログイン履歴での兆候**:
- 自分がログインしていない時間帯のログイン記録
- 見知らぬIPアドレスからのログイン
- 深夜・早朝の不審なログイン
- 短時間に複数回のログイン試行

**操作履歴での兆候**:
- 自分が実行していない操作（道具削除、スタッフ編集など）
- 権限変更の記録
- 大量のデータダウンロード
- システム設定の不正変更

**ユーザーからの報告**:
- 「勝手にパスワードが変更された」
- 「見覚えのないログイン通知が来た」
- 「自分がやっていない操作の記録がある」

---

### 3-2. 初動対応（5分以内）

#### ステップ1: 影響範囲の確認

**確認項目**:
1. **侵入されたアカウント**: 誰のアカウントか？
2. **権限レベル**: Admin / Manager / Leader / Staff
3. **侵入時刻**: いつから侵入されているか？
4. **実行された操作**: 何をされたか？

**確認方法**:
1. 右上のアイコン → **「管理者メニュー」** → **「監査ログ」**
2. **「ユーザー」** フィルタで対象ユーザーを選択
3. **「期間」** を設定（過去24時間）
4. 不審な操作を確認

---

#### ステップ2: 侵害されたアカウントの即時無効化

**手順（Admin権限者のみ実行可能）**:

1. 左サイドメニュー **「スタッフ」** をクリック
2. 侵害されたスタッフをクリック
3. **「無効化」** ボタンをクリック
4. 確認ダイアログで **「無効化する」** をクリック

**効果**:
- 即座にログアウトされる
- 再ログインできなくなる
- 進行中の不正操作を停止

---

#### ステップ3: 経営層・上司への報告

**報告先**:
- 経営層（社長、役員）
- 情報システム部門（いる場合）
- 直属の上司

**報告内容**:
```
【セキュリティインシデント報告】
発生日時: 2026年1月22日 14:30
報告者: [あなたの名前]

【インシデント内容】
- 侵害されたアカウント: 山田太郎（Admin権限）
- 侵入時刻: 2026年1月22日 02:00頃
- 不正操作: スタッフ情報の閲覧、権限変更

【初動対応】
- 14:30 アカウント無効化完了
- 14:35 本報告

【次のアクション】
- 監査ログの詳細調査
- 全スタッフのパスワード変更
- 被害範囲の特定
```

---

### 3-3. 詳細調査（30分以内）

#### 調査項目

**1. 侵入経路の特定**

監査ログで以下を確認:
- 最初のログイン時刻
- ログイン元IPアドレス
- User-Agent（ブラウザ情報）

**確認方法**:
1. 監査ログで **「アクション: ログイン」** でフィルタ
2. 侵害されたユーザーの過去7日間のログインを確認
3. 不審なIPアドレスを特定

---

**2. 実行された操作の確認**

監査ログで以下を確認:
- 閲覧されたデータ
- 変更されたデータ
- 削除されたデータ
- 権限変更
- システム設定変更

**確認方法**:
1. 監査ログで侵害されたユーザーを選択
2. 侵入時刻以降の全操作を確認
3. 各操作の内容を記録

---

**3. 被害範囲の特定**

**チェック項目**:
- [ ] 顧客情報が閲覧されたか
- [ ] スタッフ情報が変更されたか
- [ ] 権限が不正に変更されたか
- [ ] システム設定が変更されたか
- [ ] データが削除されたか
- [ ] 外部にデータが送信されたか（メール、ダウンロードなど）

---

### 3-4. 復旧対応（1時間以内）

#### ステップ1: 全スタッフのパスワード変更

**対象**:
- 侵害されたアカウントと同じ権限レベル以下の全アカウント
- Admin権限が侵害された場合 → 全員

**手順**:
1. 左サイドメニュー **「スタッフ」** をクリック
2. 各スタッフをクリック → **「パスワードリセット」**
3. 新しいパスワードを各スタッフに連絡（対面、電話）
4. 全員に「緊急のため」と理由を説明

---

#### ステップ2: 2段階認証の必須化

**手順**:
1. 右上のアイコン → **「管理者メニュー」** → **「システム設定」**
2. **「セキュリティ設定」** セクションを開く
3. **「2段階認証の必須化」** で以下をON:
   - 組織管理者（Admin）: ✅
   - マネージャー（Manager）: ✅
   - リーダー（Leader）: ✅（推奨）
   - スタッフ（Staff）: ✅（推奨）
4. **「保存」** をクリック

---

#### ステップ3: IP制限の有効化（可能な場合）

固定IPアドレスを持つ環境の場合：

**手順**:
1. 右上のアイコン → **「管理者メニュー」** → **「システム設定」**
2. **「セキュリティ設定」** セクションを開く
3. **「IPアドレス制限を有効化」** をON
4. **「許可するIPアドレス」** に自社のIPアドレスを追加
5. **「保存」** をクリック

---

#### ステップ4: 侵害されたアカウントの再有効化

侵害が収束し、安全を確認した後：

**手順**:
1. 左サイドメニュー **「スタッフ」** をクリック
2. 侵害されたスタッフをクリック
3. **「有効化」** ボタンをクリック
4. **「パスワードリセット」** をクリック
5. 新しいパスワードを本人に連絡（対面、電話）
6. 本人に2FAを必ず設定するよう指示

---

## 4. アカウント乗っ取りへの対処

### 4-1. アカウント乗っ取りの兆候

**本人からの報告**:
- 「勝手にパスワードが変更された」
- 「ログインできなくなった」
- 「見覚えのないログイン通知が来た」

**監査ログでの兆候**:
- パスワード変更の記録（本人が実行していない）
- メールアドレス変更の記録
- 2FA設定の無効化

---

### 4-2. 緊急対応（即座）

#### ステップ1: アカウント無効化

- [3-2. 初動対応 - ステップ2](#ステップ2-侵害されたアカウントの即時無効化) と同じ手順

---

#### ステップ2: 本人確認

**重要**: 報告者が本当に本人か確認する

**確認方法**:
- 対面で確認
- 社員証で確認
- 電話で確認（登録されている電話番号から）

**なりすましの可能性**:
- メールやチャットでの報告は信用しない
- 必ず対面または電話で確認

---

#### ステップ3: パスワードリセット

**手順**:
1. 左サイドメニュー **「スタッフ」** をクリック
2. 本人をクリック
3. **「パスワードリセット」** をクリック
4. 新しいパスワードを本人に伝達（対面、電話）
5. **「有効化」** をクリック

---

#### ステップ4: 2FA再設定

**手順**:
1. 本人にログインしてもらう
2. 右上のアイコン → **「アカウント設定」** → **「セキュリティ」**
3. **「2段階認証を有効化」** をクリック
4. 新しい認証方法で設定

---

### 4-3. 事後対応

**調査項目**:
- 乗っ取り期間中に何をされたか（監査ログで確認）
- 他のアカウントへの影響はないか
- データ漏洩はなかったか

**再発防止策**:
- 強固なパスワードの設定を指導
- 2FAの必須化
- パスワード使い回しの禁止を徹底

---

## 5. データ漏洩時の対応

### 5-1. データ漏洩の種類

| 種類 | 例 | 重要度 |
|------|-----|--------|
| **顧客情報** | 顧客の氏名、住所、電話番号 | 🔴 緊急 |
| **スタッフ情報** | スタッフの氏名、メールアドレス | 🟠 重大 |
| **見積書・請求書** | 金額、取引内容 | 🟠 重大 |
| **内部情報** | 道具の在庫数、現場情報 | 🟡 警戒 |

---

### 5-2. 緊急対応（即座）

#### ステップ1: 漏洩範囲の確認

**確認項目**:
- 何が漏洩したか（顧客情報、見積書など）
- いつ漏洩したか
- どこに漏洩したか（メール、USBメモリ、印刷物など）
- 誰が漏洩させたか（故意/過失）

---

#### ステップ2: 拡散防止

**対策**:
- **メール誤送信の場合**: 受信者に削除を依頼
- **SNS投稿の場合**: 即座に削除
- **USBメモリ紛失の場合**: 拾得者に返却を依頼（警察に届出）
- **印刷物紛失の場合**: 発見者に返却を依頼

---

#### ステップ3: 経営層への報告

**報告内容**:
```
【データ漏洩インシデント報告】
発生日時: 2026年1月22日 10:00
報告者: [あなたの名前]

【漏洩内容】
- 漏洩データ: 顧客情報（氏名、住所、電話番号）100件
- 漏洩経路: メール誤送信
- 送信先: 別の顧客（○○株式会社）

【初動対応】
- 10:05 受信者に削除依頼メール送信
- 10:10 受信者に電話で削除確認
- 10:15 本報告

【次のアクション】
- 顧客への謝罪と通知
- 再発防止策の策定
```

---

### 5-3. 法的対応

**個人情報保護法に基づく対応**:

1. **個人情報保護委員会への報告**（漏洩件数が一定以上の場合）
2. **本人への通知**（漏洩した個人の権利利益を害するおそれがある場合）
3. **再発防止策の策定**

**相談先**:
- 顧問弁護士
- 個人情報保護委員会（https://www.ppc.go.jp/）

## 6. パスワード漏洩時の処置

### 6-1. パスワード漏洩の種類

| 種類 | 例 | 対応 |
|------|-----|------|
| **第三者に口頭で伝えた** | 同僚に教えた | 即座にパスワード変更 |
| **メールで送信した** | 初期パスワードをメール送信 | 即座にパスワード変更 |
| **書いた紙を紛失** | パスワードメモを紛失 | 即座にパスワード変更 |
| **フィッシング詐欺** | 偽サイトに入力 | 即座にパスワード変更 + 監査ログ確認 |
| **他サービスでの漏洩** | Gmailのパスワードを使い回し | ザイロクのパスワードも変更 |

---

### 6-2. 緊急対応（即座）

#### ステップ1: パスワード変更

**本人が実行**:
1. ログイン
2. 右上のアイコン → **「アカウント設定」** → **「セキュリティ」**
3. **「パスワード変更」** をクリック
4. 新しいパスワードを設定

**Admin権限者が実行**（本人がログインできない場合）:
1. 左サイドメニュー **「スタッフ」** をクリック
2. 本人をクリック
3. **「パスワードリセット」** をクリック
4. 新しいパスワードを本人に伝達（対面、電話）

---

#### ステップ2: 監査ログ確認

**確認項目**:
- パスワード漏洩後に不正ログインがなかったか
- 不正な操作がなかったか

**確認方法**:
1. 右上のアイコン → **「管理者メニュー」** → **「監査ログ」**
2. 本人のアカウントで漏洩時刻以降のログを確認
3. 不審なログインや操作がないか確認

**不正ログインが確認された場合**:
- [3. 不正アクセス検知時の対応](#3-不正アクセス検知時の対応) を参照

---

#### ステップ3: 2FA設定

パスワード変更後、2FAを設定：

**手順**:
1. 右上のアイコン → **「アカウント設定」** → **「セキュリティ」**
2. **「2段階認証を有効化」** をクリック
3. 認証方式を選択（認証アプリ推奨）
4. 設定を完了

## 7. システム全体への影響がある場合

### 7-1. ランサムウェア感染

**症状**:
- ファイルが開けない
- 「ファイルが暗号化されました」というメッセージ
- 身代金を要求される

**対応**:
1. **即座にネットワークから切断**（Wi-FiをOFF、LANケーブルを抜く）
2. **他のPCを使用して経営層・システム管理者に報告**
3. **警察に通報**（サイバー犯罪相談窓口）
4. **専門業者に相談**（セキュリティベンダー）

**注意**: 身代金を支払わない（支払ってもファイルが戻る保証はない）

---

### 7-2. サービス全体の停止

**症状**:
- ザイロクに全員がアクセスできない
- 「サーバーエラー」と表示される

**対応**:
1. **システム管理者に連絡**
2. **他社サービス（Supabase）のステータスページを確認**
3. **経営層に報告**
4. **復旧を待つ**

**ユーザーができること**:
- 業務を手動（紙、Excel）で継続
- 復旧後にデータを入力

---

## 8. 事後対応とレビュー

### 8-1. インシデント報告書の作成

インシデント収束後、詳細な報告書を作成：

**報告書の項目**:
1. **インシデント概要**（何が起きたか）
2. **発生日時**
3. **発見日時**
4. **発見方法**（監査ログ、ユーザー報告など）
5. **影響範囲**（被害を受けたアカウント、データ）
6. **原因**（パスワードの使い回し、フィッシングなど）
7. **対応履歴**（タイムライン）
8. **再発防止策**

---

### 8-2. 再発防止策の実施

**技術的対策**:
- [ ] 2FA必須化
- [ ] IP制限の有効化
- [ ] パスワードポリシーの強化

**運用的対策**:
- [ ] セキュリティ教育の実施
- [ ] 監査ログの定期確認
- [ ] インシデント対応訓練

**組織的対策**:
- [ ] セキュリティポリシーの策定
- [ ] インシデント対応体制の整備
- [ ] 責任者の明確化

---

### 8-3. 全員へのフィードバック

インシデントから学んだ教訓を全員に共有：

**朝礼での周知例**:
```
【セキュリティインシデント共有】
先週、不正ログインが発生しました。

【原因】
・パスワードの使い回し
・2FAが未設定

【対策】
・全員、今日中にパスワードを変更してください
・2FAを必ず設定してください
・パスワードは他のサービスと使い回さないでください

【質問があれば】
Admin権限者（○○）まで
```
